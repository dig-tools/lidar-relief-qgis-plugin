# From Visualization to Discovery: A Strategic Roadmap for the LiDAR Relief Plugin v2.0

## Multi-Temporal Analysis and Advanced Change Detection

The analysis of repeat LiDAR surveys offers a powerful method for monitoring archaeological landscapes, enabling the detection of subtle changes caused by natural processes, agricultural activity, or human intervention such as looting [[24](https://arxiv.org/html/2409.09432v1), [52](https://www.researchgate.net/publication/228970086_Change_detection_of_landslide_terrains_using_ground-based_LiDAR_data)]. The foundational technique for this type of analysis is the creation of a Differential Digital Elevation Model (dDEM), which involves subtracting one Digital Elevation Model (DEM) from another to produce a map highlighting topographic change [[36](https://www.mdpi.com/2072-4292/13/13/2629)]. This method has been successfully applied to detect changes in landslide terrains and urban environments, demonstrating its utility for monitoring purposes [[36](https://www.mdpi.com/2072-4292/13/13/2629), [52](https://www.researchgate.net/publication/228970086_Change_detection_of_landslide_terrains_using_ground-based_LiDAR_data)]. Research indicates that even when surveys are not perfectly co-registered, advanced merging techniques can be used to create reliable dDEMs, making this approach robust for analyzing historical datasets [[41](https://www.mdpi.com/2073-445X/11/11/1964)]. The availability of national, multi-temporal LiDAR programs, such as the UK Environment Agency's initiative which captured elevation data for all of England at 1m resolution over multiple winters from 2017 to 2022, provides a rich source of data for such analyses [[83](https://experience.arcgis.com/experience/753ad2ebd3554fa696885b8c366c3049/page/Open-Data)]. Beyond simple subtraction, more sophisticated machine learning approaches have been developed, often involving hand-crafted features to classify changes in urban point clouds [[36](https://www.mdpi.com/2072-4292/13/13/2629)]. Furthermore, time-series analysis of satellite imagery can identify crop-mark anomalies that may correlate with physical changes detected by LiDAR, offering a complementary analytical pathway [[24](https://arxiv.org/html/2409.09432v1), [87](https://www.mdpi.com/2504-446X/4/4/72)].

For implementation within the QGIS plugin, several options exist with varying levels of risk and impact. A low-risk implementation would be to add a new "Difference" algorithm that performs a direct raster subtraction of two input DEMs. This would extend the plugin's existing NumPy-based architecture and provide immediate value. A medium-risk but highly beneficial enhancement would be to integrate Python libraries like `xarray` or `rioxarray`. These libraries are specifically designed for handling multi-dimensional arrays and are ideal for efficiently managing and processing raster time series [[16](https://www.linkedin.com/in/benoit-parmentier-3413423a), [49](https://uwgda-jupyterbook.readthedocs.io/en/latest/modules/07_Raster2_DEMs_Warp_Clip_Sample/07_Raster2_DEMs_Warp_Clip_Sample_demo.html)]. They simplify critical operations such as aligning DEMs from different dates, calculating statistics across the time dimension, and managing complex geospatial metadata, thereby increasing the robustness and reliability of the change detection workflow [[114](https://arxiv.org/html/2411.11291v1)]. High-risk, research-stage implementations involving fully automated ML classification of change types are less suitable for a v2.0 release but could be supported by the plugin acting as a pre-processing tool to generate dDEMs for external analysis. The primary risks involve the complexity of integrating new dependencies like `xarray` and the potential for increased memory usage when handling large multi-temporal datasets. However, given the growing availability of public multi-temporal LiDAR data, implementing this capability is a high-priority feature that directly addresses a significant analytical need in modern archaeology, transforming the plugin from a static visualization tool into a dynamic analysis platform.

| Feature Area | Recommended Implementation Path | Technical Risk | Priority |
| :--- | :--- | :--- | :--- |
| **Basic Difference Calculation** | New plugin algorithm using existing NumPy/GDAL core [[36](https://www.mdpi.com/2072-4292/13/13/2629)]. | Low | High |
| **Multi-Temporal Workflow** | Integrate `xarray` or `rioxarray` for robust time-series handling [[16](https://www.linkedin.com/in/benoit-parmentier-3413423a), [49](https://uwgda-jupyterbook.readthedocs.io/en/latest/modules/07_Raster2_DEMs_Warp_Clip_Sample/07_Raster2_DEMs_Warp_Clip_Sample_demo.html)]. | Medium | High |
| **Advanced ML Classification** | Provide dDEM output for use in external ML tools; avoid building models internally. | High | Low (Future Research) |

## In-Plugin Artificial Intelligence and Machine Learning

The integration of artificial intelligence (AI) and machine learning (ML) represents a significant opportunity to enhance the plugin's capabilities beyond traditional visualization. Research demonstrates the successful application of deep learning models to detect a variety of archaeological features. For instance, projects are underway to use AI to create a national ridge-and-furrow dataset for England [[76](https://www.researchgate.net/publication/374000171_Project_G_Using_artificial_intelligence_to_create_a_national_ridge_and_furrow_dataset_for_England)], while other studies have employed object detection methods to identify moated sites from satellite imagery [[100](https://www.sciencedirect.com/science/article/abs/pii/S0305440324001389)]. A key enabler for practical integration into a QGIS plugin is the use of the Open Neural Network Exchange (ONNX) format, which allows models trained in frameworks like PyTorch or TensorFlow to be deployed efficiently [[71](https://arxiv.org/pdf/2605.26119), [75](https://www.cambridge.org/core/journals/journal-of-plasma-physics/article/towards-fast-and-accurate-predictions-of-radio-frequency-power-deposition-and-current-profile-via-datadriven-modelling-applications-to-lower-hybrid-current-drive/E3D8D54FE39896E03C5DCDBCE9300297)]. For CPU-based inference, the combination of ONNX Runtime with the Intel OpenVINO Execution Provider can deliver substantial performance improvements, with reported speed-ups of up to 10x on CPUs compared to standard runtimes [[58](https://www.linkedin.com/pulse/benchmarking-openvino-accelerating-onnx-inference-jerome-revillard-gxhie), [59](https://stackoverflow.com/questions/70844974/onnxruntime-vs-onnxruntimeopenvinoep-inference-time-difference)]. Lightweight model architectures such as MobileNet-based YOLO variants are particularly well-suited for this context, offering a good balance between detection accuracy and fast inference times on a standard processor [[73](https://www.researchgate.net/publication/381961120_YOLOv5_YOLOv8_and_YOLOv10_The_Go-To_Detectors_for_Real-time_Vision), [74](https://www.mdpi.com/2077-1312/13/5/947)]. While no single, universally accepted pre-trained model for all archaeological features currently exists, active projects are developing such resources, and open training datasets may become available from heritage agencies [[76](https://www.researchgate.net/publication/374000171_Project_G_Using_artificial_intelligence_to_create_a_national_ridge_and_furrow_dataset_for_England), [92](https://www.researchgate.net/publication/371166500_Project_D_Assessing_the_use_of_artificial_intelligence_to_identify_and_record_ridge_and_furrow_earthworks)].

For the plugin, a practical implementation strategy would treat AI as an optional component rather than a core function. A low-risk approach would be to develop an "AI Detection" processing tool that acts as an inference wrapper. This would not involve training models itself but would allow users to leverage their own pre-trained ONNX models. A medium-risk implementation would involve adding the `onnxruntime-openvino` library as an optional dependency [[57](https://pypi.org/project/onnxruntime-openvino/)]. The plugin could then provide a simple interface for loading a user-provided `.onnx` model file and a corresponding JSON file defining the class labels. This modular approach keeps the core plugin lightweight while providing access to state-of-the-art detection capabilities. The primary technical risk lies in managing the additional software dependencies, although this is a common practice for advanced QGIS plugins. The main uncertainty is the long-term availability of truly open, high-quality training datasets for diverse archaeological features. Nonetheless, prioritizing this feature is strategically important. It would position the plugin as a forward-looking, state-of-the-art prospection tool, directly supporting both automated discovery and expert-guided interpretation workflows.

## Direct Point Cloud Processing and Archaeological Analytics

Expanding the plugin's functionality to include direct processing of LAS/LAZ files represents a major step forward in streamlining archaeological workflows. Standard LiDAR filtering algorithms can often misclassify subtle archaeological earthworks as noise, leading to poor quality Digital Terrain Models (DTMs) [[65](https://www.mdpi.com/2072-4292/13/9/1855)]. To address this, specialized ground filtering algorithms are essential. The Cloth Simulation Filter (CSF) is a prominent example, designed specifically for complex terrain and proven effective in archaeological contexts [[18](https://pypi.org/project/cloth-simulation-filter/)]. Crucially, a Python package named `cloth-simulation-filter` is available via pip, which simplifies its integration into the plugin without requiring extensive custom development [[18](https://pypi.org/project/cloth-simulation-filter/)]. Another avenue for enhanced analysis is the use of full-waveform LiDAR (FWF) data, which captures the entire return of the laser pulse and can provide more detailed information about vegetation structure and ground surface [[32](https://www.sciencedirect.com/science/article/abs/pii/S0305440307001252), [33](https://www.researchgate.net/publication/254593934_Full-waveform_airborne_laser_scanning_as_a_tool_for_archaeological_reconnaissance)]. Similarly, the radiometrically calibrated intensity data from sensors like single-photon LiDAR (SPL) may offer additional clues for feature detection, though this remains an emerging research area [[34](https://www.mdpi.com/2072-4292/11/8/945)]. Several Python libraries exist for point cloud manipulation, including `laspy`, PDAL, and Open3D, which could be integrated as optional dependencies to enable this functionality [[43](https://menvuthy.readthedocs.io/en/latest/Content/Documentation/lidar/Create_DEM_and_Hillshade_from_point_cloud_data_in_Python.html), [67](https://www.researchgate.net/post/Does-anybody-know-an-open-source-LiDAR-processing-software)].

The implementation path for these features carries a medium level of risk due to the introduction of new dependencies. A low-risk starting point would be to implement the CSF algorithm as a standalone tool within the Processing Toolbox. This would allow users to generate a superior DTM directly from a raw point cloud, bypassing the need for intermediate conversion steps. A medium-risk extension would be to develop a more comprehensive pipeline that uses `laspy` or PDAL to read LAS/LAZ files directly, allowing the plugin's existing visualization algorithms to operate on the point cloud data before generating a final raster product [[43](https://menvuthy.readthedocs.io/en/latest/Content/Documentation/lidar/Create_DEM_and_Hillshade_from_point_cloud_data_in_Python.html)]. This would significantly streamline the workflow for users working with raw LiDAR data. Implementing algorithms that specifically exploit full-waveform or intensity data is a high-risk, research-intensive task that should be deferred until more mature open-source tools and validated methodologies become available. Given that supporting LAS/LAZ input was already identified in the plugin's roadmap, prioritizing this work is crucial for maintaining relevance and utility in modern archaeological practice [[66](https://www.researchgate.net/publication/353929383_Airborne_LiDAR_Point_Cloud_Processing_for_Archaeology_Pipeline_and_QGIS_Toolbox)]. The ability to perform high-quality, archaeology-specific ground filtering directly within QGIS would be a highly valued feature.

## Automated Workflows for Field Integration and Reporting

Creating seamless workflows that connect desktop analysis with fieldwork and project documentation is critical for the plugin's adoption. The de facto standard for open-source mobile GIS in archaeology is QField, the mobile counterpart of QGIS, often synchronized via platforms like Mergin Maps [[11](https://www.sciencedirect.com/science/article/pii/S2590197424000442), [95](https://zenodo.org/records/14598657)]. Documented workflows now exist for using mobile applications to conduct ground-truthing based on LiDAR-derived visualizations and to record observations in the field [[77](https://zenodo.org/doi/10.5281/zenodo.8303247), [94](https://www.researchgate.net/publication/404120030_Crossing_software_boundaries_From_iDig_and_ODK_to_QField_for_archaeological_field_documentation)]. A low-risk implementation would be to add an "Export Detected Features" tool that saves point geometries representing anomalies to a structured vector format like GeoPackage [[14](https://docs.qgis.org/latest/en/docs/user_manual/managing_data_source/create_layers.html)]. GeoPackage is ideal as it can store both geometry and attributes, making it easily loadable into QField for inspection and annotation. Enhancing this tool to include relevant metadata—such as a feature ID, detection confidence score, or timestamp—as attributes would further enrich the data available during fieldwork. Another highly valuable feature is automated report generation. Archaeological reports must adhere to strict standards, such as those outlined by the Chartered Institute for Archaeologists (CIfA) [[1](https://www.researchgate.net/publication/371390470_2022_CIfA_International_cultural_heritage_practice_competence_matrix_-_National_Occupational_Standards_in_Archaeological_Practice), [42](https://www.scribd.com/document/329841373/CIfAS-GFieldevaluation-1-pdf)]. Python offers several libraries for generating PDFs; WeasyPrint is often preferred for its ease of use and ability to convert HTML/CSS layouts into publication-quality documents, while ReportLab offers more granular control for script-based generation [[4](https://blog.csdn.net/2501_91115965/article/details/146712907), [90](https://zhuanlan.zhihu.com/p/565835466)]. A low-risk implementation would involve creating a "Generate Report" button that uses one of these libraries to compile a document containing essential information: input DEM metadata, processing parameters, visualizations, and a list of detected features. Including compliance with geospatial metadata standards like ISO 19115 would further enhance the professionalism of the output [[50](https://lobehub.com/skills/k-dense-ai-claude-scientific-skills-geomaster)]. Both of these features carry low technical risk and represent a high priority, as they address critical gaps in the archaeological investigation lifecycle, solidifying the plugin's role as an indispensable tool from analysis to delivery.

## Strategic Positioning and Competitive Differentiation

To ensure the plugin's long-term relevance and indispensability, it must evolve beyond a collection of visualization algorithms into an integrated digital prospection platform. The competitive landscape includes other QGIS plugins like the Open LiDAR Toolbox and the Relief Visualisation Toolbox, as well as broader commercial platforms such as ArcGIS Pro extensions and web-based solutions like Google Earth Engine [[46](https://www.researchgate.net/publication/371226392_QGIS_Relief_Visualisation_Toolbox_plugin_AARG_Online_workshop_25_May_2023), [62](https://www.researchgate.net/publication/354237142_Open_Lidar_Toolbox_Qgis_Plugin_httpspluginsqgisorgpluginsopen_lidar_tools), [102](https://www.researchgate.net/publication/337153388_Google_Earth_as_a_Powerful_Tool_for_Archaeological_and_Cultural_Heritage_Applications_A_Review)]. The wider trend in archaeological remote sensing is toward integrated, AI-driven, and cloud-powered ecosystems [[12](https://www.linkedin.com/posts/kanchan-ai-product-builder_ai-ai-ml-activity-7398428528863514624-NDn-)]. Standalone software like CloudCompare remains popular for pure point cloud editing, but often lacks the advanced analytical and reporting functionalities desired in an archaeological context [[67](https://www.researchgate.net/post/Does-anybody-know-an-open-source-LiDAR-processing-software)]. The plugin's key differentiator is its deep focus on relief visualization combined with its seamless integration into the familiar and powerful QGIS environment. By implementing the proposed v2.0 features—particularly in-plugin AI detection, multi-sensor fusion, and automated reporting—the plugin can offer a complete, end-to-end workflow that competitors may not match. This strategy positions the plugin as the premier open-source alternative to proprietary, closed ecosystems, appealing strongly to academic researchers and heritage management bodies that prioritize open data and collaborative development.

To achieve this, the plugin must solve critical workflow bottlenecks. The connection between digital analysis (in QGIS) and physical ground-truthing (via QField export) closes a vital loop in the investigative process [[77](https://zenodo.org/doi/10.5281/zenodo.8303247), [94](https://www.researchgate.net/publication/404120030_Crossing_software_boundaries_From_iDig_and_ODK_to_QField_for_archaeological_field_documentation)]. The ability to automatically generate standards-compliant PDF reports and publish interactive web maps transforms the plugin from a discovery tool into a comprehensive project management system [[1](https://www.researchgate.net/publication/371390470_2022_CIfA_International_cultural_heritage_practice_competence_matrix_-_National_Occupational_Standards_in_Archaeological_Practice), [13](https://www.linkedin.com/posts/adnangispro_datascience-geospatial-gis-activity-7362781489886818304-qW35)]. The following table summarizes the strategic priorities for v2.0:

| Feature Category | Key Features | Strategic Impact | Priority |
| :--- | :--- | :--- | :--- |
| **Workflow Automation** | Automated PDF Report Generation, Visualization Recipe Sharing, Field Export (GeoPackage). | Reduces manual effort, standardizes outputs, connects analysis to fieldwork. | High |
| **Analytical Power** | In-Plugin AI/ML Detection, Multi-Sensor Fusion (LiDAR + Multispectral), Multi-Temporal Change Detection. | Elevates plugin from visualization to discovery engine; broadens problem-solving scope. | High |
| **Data Capabilities** | Direct LAS/LAZ Input, Support for Full-Waveform Data (future). | Streamlines workflow, improves data quality, increases utility for raw data users. | High |
| **Scalability & Performance** | Tiled Processing, CuPy GPU Acceleration (long-term). | Ensures usability with very large datasets; future-proofs against growing data volumes. | Medium |

GPU acceleration for computationally intensive algorithms like SVF and ASVF presents a high-risk, high-effort challenge. While CuPy offers a NumPy-compatible API for GPU computing, rewriting and validating these algorithms is a significant undertaking [[84](https://forums.developer.nvidia.com/t/gdal-wont-run-on-gpu-with-cuda-or-opencl-forced-to-cpu/36986)]. This should be considered a long-term goal for v2.1 or beyond, as the plugin's current tiled processing is already effective for many large datasets. A pragmatic v2.0 plan should focus on the high-impact, lower-risk features that collectively transform the plugin into an indispensable hub for archaeological geospatial analysis.
