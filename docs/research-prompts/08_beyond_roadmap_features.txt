Project: LiDAR-Relief-QGIS-Plugin
Source: Future feature exploration — beyond the v1.3 roadmap
======================================================================

CONTEXT:
I maintain the LiDAR Relief Visualization QGIS plugin (15 algorithms:
Multi-Hillshade, SLRM, SVF, Slope, Openness, MSTP, VAT, Simple Red
Relief, Blend, Local Dominance, ASVF, e4MSTP, PCA Composite, ML-Ready
Export, Batch mode). The plugin is at v1.3.5, published on the QGIS
Plugin Repository, with ~2,500 users.

Current architecture: pure NumPy/GDAL core algorithms with thin QGIS
Processing wrappers. Tiled processing for large DEMs. All algorithms
are tested with pytest. The plugin is GPL v2 licensed.

The existing ROADMAP.md (docs/ROADMAP.md) covers v1.0 completed work,
v1.1–v1.3 delivered features, and a brief "Deferred / Future Research"
section mentioning GPU acceleration, direct DEM download, LAS/LAZ
input, and geomorphon classification.

I now need to look BEYOND the current roadmap for the next generation
of features. Please research and provide detailed, evidence-based
answers.

──────────────────────────────────────────────────────────────────────

DEEP RESEARCH PROMPT:

I am evaluating what new capabilities to add to this QGIS LiDAR relief
visualization plugin in its next major development cycle (v2.0+).
The plugin already implements the full RVT algorithm suite plus batch
processing. I need to understand:

1. MULTI-TEMPORAL LiDAR CHANGE DETECTION

   - What published methods exist for detecting archaeological features
     by comparing repeat LiDAR surveys of the same area?
   - The UK Environment Agency has flown national LiDAR multiple times
     (2000s, 2017–2022). What algorithms detect newly revealed features
     (differential DEM, coherence maps, machine learning approaches)?
   - What are the computational and storage implications of maintaining
     time-series DEMs in a QGIS plugin?
   - Are there off-the-shelf Python libraries (e.g., xarray, rioxarray)
     that simplify multi-temporal raster workflows?

2. IN-PLUGIN AI/ML FEATURE DETECTION

   - What lightweight ONNX / OpenVINO models exist for archaeological
     feature detection in LiDAR relief visualizations?
   - Has anyone published a pre-trained model that runs on CPU in
     real-time (or near-real-time) for detecting barrows, ring ditches,
     ridge-and-furrow, or charcoal kilns?
   - What are the most appropriate model architectures for QGIS
     integration (U-Net, YOLO, Mask R-CNN) in terms of accuracy vs
     CPU inference speed?
   - What open training datasets exist (e.g., from Historic England,
     ARCHES, national heritage agencies)?
   - Would an AI detection feature be practical to integrate as an
     optional plugin component, or would it require a separate
     companion application?

3. ARCHAEOLOGY-SPECIFIC POINT CLOUD PROCESSING

   - Beyond simple ground classification, what LiDAR point cloud
     analytics are specific to archaeology?
   - What published research exists on using full-waveform LiDAR
     returns for archaeological prospection?
   - Can intensity return information from single-photon LiDAR
     (SPL) or Geiger-mode LiDAR be exploited for archaeological
     feature detection?
   - Are there Python libraries (PDAL, laspy, open3d) that could
     be integrated as optional plugin dependencies for direct
     point cloud processing?
   - What ground filtering algorithms are specifically optimized for
     archaeological terrain (preserving subtle earthworks that standard
     filters classify as noise)?

4. CLOUD-OPTIMIZED GEOTIFF AND WEB PUBLISHING

   - What is the current best practice for generating web-ready
     visualizations from LiDAR data (COGs, XYZ tiles, MBTiles)?
   - Could the plugin generate shareable web maps (e.g., GitHub Pages
     with Leaflet/MapLibre) from a DEM and a set of visualizations?
   - Are there lightweight Python libraries (rio-cogeo, rasterio) that
     could handle the conversion?
   - What file size ranges are realistic for publishing a typical
     county-scale LiDAR survey (5,000–50,000 km²) as an interactive
     web map?

5. FIELD SURVEY INTEGRATION AND MOBILE WORKFLOW

   - How could the plugin integrate with field survey tools like
     QField, Mergin, or Input?
   - What is the best format for exporting detected anomaly locations
     for ground-truthing (GeoPackage, GPX, KML with structured
     attribute schemas)?
   - Are there published smartphone/tablet workflows for in-field
     inspection of LiDAR relief visualizations?
   - Could the plugin generate printable field maps or A3 survey
     sheets automatically?

6. FUSION WITH MULTISPECTRAL / SATELLITE IMAGERY

   - What published research exists on fusing LiDAR-derived relief
     with multispectral satellite imagery (Sentinel-2, Planet) or
     aerial photography for improved archaeological prospection?
   - Are there established band-combination / blend recipes (similar
     to e4MSTP's composite build order) for LiDAR + multispectral
     fusion?
   - Could the plugin load and co-register satellite imagery as
     additional input layers?
   - What are the radiometric calibration requirements for correct
     multi-sensor fusion?

7. AUTOMATED REPORT GENERATION

   - What is the standard format for archaeological geophysical /
     remote sensing survey reports (e.g., UK Chartered Institute for
     Archaeologists standards)?
   - Could the plugin auto-generate a PDF report containing: input DEM
     metadata, visualizations, processing parameters, coordinate
     reference system information, and detected anomaly locations?
   - Are there Python libraries (ReportLab, WeasyPrint, LaTeX) suitable
     for generating publication-quality reports from QGIS plugin
     output?
   - What metadata standards apply (ISO 19115, INSPIRE, MIDAS Heritage)?

8. PERFORMANCE AND SCALABILITY ADVANCEMENTS

   - How effective is CuPy/CUDA GPU acceleration for the horizon-
     scanning algorithms (SVF, ASVF, Openness, Local Dominance)?
   - Are there published benchmarks comparing NumPy vs CuPy performance
     for these specific algorithms?
   - What is the practical maximum DEM size that a QGIS plugin can
     process with current consumer hardware (16GB, 32GB RAM)?
   - Could the plugin offer a "cloud processing" mode where large
     DEMs are sent to a server for processing?

9. USER COMMUNITY AND SHARING

   - Are there successful examples of QGIS plugins with built-in
     sharing/recipe exchange (e.g., parameter preset sharing via
     GitHub Gist or QGIS Plugin Repository)?
   - What would a "Visualization Recipe" format look like — a JSON
     file capturing all parameters needed to reproduce a specific
     visualization on any DEM?
   - Could the plugin support user-contributed landscape presets
     beyond the four built-in presets (Flat, Forested, Upland,
     Coastal)?

10. COMPETITIVE LANDSCAPE ECOSYSTEM MATURITY

    - Are there new standalone LiDAR visualization tools or QGIS
      plugins that have emerged since 2023 that I should be aware of?
    - What features do they offer that this plugin does not?
    - Which direction is the wider field of archaeological remote
      sensing heading (automated detection, multi-sensor fusion,
      open data, cloud processing)?
    - What would make this plugin indispensable rather than just
      convenient?

For each research area, please provide:
- A summary of the current state of the art in 1–2 paragraphs
- Concrete implementation recommendations (what is practical to
  integrate into a QGIS plugin within 3–6 months)
- Risk assessment: high risk (research-stage), medium risk (requires
  new dependencies), low risk (extension of existing patterns)
- Priority ranking for a v2.0 planning cycle
